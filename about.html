• Why?

    The phylotastic project aims to enable flexible, automated access to expert knowledge of the "Tree of Life" of millions of species. Given a phylogenetic framework that covers most or all species, a user ought to be able to get a subtree tailored to a particular research problem (e.g., a phylogeny of "model organisms") but currently this isn't possible (except in the case of plants, via Phylomatic)

  • What?

    Rather than building a single centralized system, we want to promote a design based on loosely coupled components that can be configured so as to solve various problems. To discourage the idea of a single centralized system, we like to use "phylotastic" as an adjective, not a noun. A system is phylotastic if it meets certain criteria. A performance-based definition might look like this:

    An application is phylotastic to the extent that, in response to a user query S, it supplies expert phylogenetic knowledge of S, in a form useful for research purposes, doing so in a timely manner.

    where S is a list of identifiers such as { "Mus musculus", "Homo sapiens", "Pan troglogytes" }. Note some aspects of this definition:

    • User-oriented - responds to user query
    • Expert knowledge â€“ exposes expert knowledge to user who is not an expert
    • Research-ready - provides a robust computable tree with provenance
    • Timely - provides results quickly enough to support high-volume use-cases
